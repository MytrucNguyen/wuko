# design-sync notes — Wuko

Repo-specific gotchas for future syncs. Append a bullet whenever you learn something.

## Repo shape
- Wuko is distributed via the **shadcn registry** model — components are **source `.tsx`** under `apps/docs/registry/default/ui/`, with **no published `dist/`**. There is no conventional DS package.
- A **synthetic barrel** at `.design-sync/pkg/index.ts` (package name `wuko`) re-exports every registry component as the bundle entry. `cfg.entry` points at it; `cfg.pkg`=`wuko`, `cfg.globalName`=`Wuko`. This is the durable bundle entry — regenerate it if components are added/removed under `registry/default/ui/`.
- Source shape is **storybook** (`apps/docs/.storybook`, `@storybook/nextjs-vite`). Stories live in `apps/docs/stories/*.stories.tsx`, importing components via the `@/registry/default/ui/*` alias.
- `cfg.tsconfig` = `../../apps/docs/tsconfig.json` — needed so esbuild resolves the `@/*` and `@/components/ui/*` path aliases in both the bundle build and preview compiles. **Note: `cfg.tsconfig` (and the other `cfg.*` path fields) are resolved relative to `PKG_DIR` (`.design-sync/pkg`), NOT cwd** — hence the `../../` prefix. (`cfg.entry` / `cfg.storybookStatic` / `cfg.storybookConfigDir` are cwd-relative; pass `--entry .design-sync/pkg/index.ts` on the CLI too.)
- **Symlink `.design-sync/pkg/node_modules -> ../../apps/docs/node_modules`** so ts-morph resolves `@types/react` (and the radix/etc. deps) from the synthetic package during `.d.ts` extraction. Gitignored + per-clone — recreate on a fresh clone: `ln -sfn ../../apps/docs/node_modules .design-sync/pkg/node_modules`. Without it the build warns `[DTS_REACT]` and `ref` types lose their type param.
- Build/validate invocation: `--node-modules apps/docs/node_modules --entry .design-sync/pkg/index.ts`.
- `cfg.storyImports.shim = ["/registry/default/ui/"]` forces story component imports to resolve to `window.Wuko.*` (the shipped bundle) instead of bundling a fresh source copy — so a passing compare verifies the actual bundle, not just the source. All stories use **named** imports, which the root-namespace shim resolves via `export *`.

## Styling (Tailwind v4)
- Components are styled with **Tailwind v4 utility classes** + `--wuko-*` CSS variable tokens (see `apps/docs/app/globals.css`, `THEMING.md`). The utilities are compiled by Tailwind at build time, so CSS must come from the **compiled Storybook** (`[CSS_FROM_STORYBOOK]` scrape of `sb-reference`), NOT the raw `globals.css` (which is Tailwind source — do not set `cfg.cssEntry` to it).
- **Theme: dark is the default** — dark tokens are defined on `:root`; `[data-theme="light"]` overrides. There is no `[data-theme="dark"]` selector, so previews render dark with no attribute set, matching the Storybook reference (`initialGlobals.theme = "dark"`).

## Grading learnings
- [GENERAL] **Framing differs by design** — Storybook stories use `layout: "centered"` and crop tight around the component; preview cards render the component top-left in a full-viewport dark page. Judge the component, not the surrounding whitespace/scale. The compare sheet shrinks both panels to fit, exaggerating the apparent size gap; open the `raw/` PNGs to confirm the component itself is identical.
- [GENERAL] Icons (lucide-react) bundle and render correctly in previews (verified on Button leading/trailing/icon-only stories). `lucide-react` is bundled into the preview from source (not shimmed) since it's not the DS package.
- [GENERAL] Owned previews must import components via a **relative path** (`../../apps/docs/registry/default/ui/<name>`), NOT the `@/` alias — esbuild's tsconfig auto-discovery doesn't reach `.design-sync/previews/`. The relative path still gets shimmed to `window.Wuko` via `cfg.storyImports.shim` (`/registry/default/ui/`).
- Card: owned preview (`.design-sync/previews/Card.tsx`) re-implements Default/Stat/WithAction; the `WithImage` story uses `next/image` (crashes with `process is not defined` at module import) + a 1.5MB local asset, so it is skipped (`cfg.overrides.Card.skip`) and omitted from the owned preview.
- Skeleton, Pagination: `cardMode: "column"` — their wide stories (Table / Default) overflow a grid cell.
- [GENERAL] **Preview backdrop is navy, fixed via the `css-fallback.mjs` fork.** The preview HTML template (`emit.mjs`, app-contract — not forkable) hardcodes `<style>body{background:#fff}</style>`, which overrides the bundle's `html,body{background:var(--wuko-bg)}`. For a dark-default DS this washes out every TRANSLUCENT/transparent surface (`bg-wuko-card/70`, `bg-*/15`, 50%-opacity disabled controls, transparent table cells) — they composite over white instead of navy, so Card/Skeleton/DataTable/Select-disabled all rendered too light vs Storybook (which is navy). The fork appends `html body{background:var(--wuko-bg)}` (specificity 0,0,2 beats the inline `body` 0,0,1) to the scraped `_ds_bundle.css`, so the preview — and real designs — render on navy. This also fixes the "silver disabled control" symptom (it's `bg-wuko-card` @ opacity .5; over navy it goes dark). `color-scheme:dark` is already in the bundle `:root` and was NOT the cause.
- Avatar: `WithImage` story uses a 1.5MB local static asset (`/brand/wuko.png`) not shipped in the bundle, so the preview 404s and Avatar correctly falls back to initials (component is intact). Skipped via `cfg.overrides.Avatar.skip` — image-avatar support is verified by the graceful fallback; the demo asset isn't worth shipping.

## Re-sync risks
- **Fonts load from Google Fonts CDN at runtime** (`[FONT_REMOTE]`): styles.css `@import`s the Google Fonts URL and the `css-fallback.mjs` fork defines `--font-inter`/`--font-jetbrains`. If Wuko ever self-hosts fonts or the preview-head changes, re-verify the fork still harvests the right vars (watch for `[FONT_VARS_FROM_PREVIEW_HEAD]` in the build log; missing → fonts regress to UA default).
- The `css-fallback.mjs` fork harvests `--font-*` vars from `sb-reference/iframe.html` inline `<style>`. If the storybook `preview-head.html` stops defining them inline, the harvest yields nothing — re-check after any storybook/preview-head change.
- Card owned preview is tied to the current Card story JSX — if `card.stories.tsx` changes (`[STORY_CHANGED]`), update `.design-sync/previews/Card.tsx` to match.
- Button captured all 17 stories (cap raised); other components capture the first 6 by default ([STORY_CAP]) — tail stories are verified-by-upload, not individually graded.
- `--shiki-light`/`--shiki-dark` are referenced by scraped code-block CSS but undefined (below the missing-token threshold); no registry component uses them, so this is expected.
- `cfg.guidelinesGlob: ["../../THEMING.md"]` ships the repo-root `THEMING.md` into `guidelines/` so the conventions header can point the design agent at the canonical token table. If `THEMING.md` is renamed/moved, update the glob (and the `conventions.md` reference to `guidelines/THEMING.md`).
- Select has NO owned preview (a subagent's `color-scheme` workaround was deleted once the navy-backdrop fork fixed the disabled-control root cause). Its preview is generated; the trigger renders full-width in the card because `SelectTrigger` is `w-full` — this is correct container behavior (graded match), not a defect. Don't "fix" it with an owned width frame.
- Decorator bundling always fails with `Could not resolve "tailwindcss"` (preview.ts imports `../app/globals.css` which `@import`s tailwindcss). Harmless: the only thing the decorator did was set `data-theme="dark"`, which equals the `:root` default, so previews render dark without it. No `cfg.provider` needed.
